## Part 1: Line and Area Charts

### What is a Line Chart in D3?
![[Pasted image 20260401005842.png]]
A **line chart** in D3 is built by initialising a line generator (`d3.line()`), configuring its x and y accessor functions using scales, then appending a single SVG `<path>` element whose `d` attribute is set by calling the generator on the dataset.

In simple terms: you tell D3 where each point sits horizontally and vertically, and D3 draws a connected path through all of them.

Think of it like: giving D3 a list of GPS coordinates and asking it to trace the route on a map. The scales are the GPS conversion layer.

Why it matters: line charts are the standard tool for continuous time-series data such as temperature, stock prices, or energy consumption.

### The Margin Convention
![[Pasted image 20260401005917.png]]
Before appending anything, D3 projects define a **margin convention**: an object with top, right, bottom, and left values that reserves whitespace around the inner chart for axes, labels, and legends.
```js
const margin = { top: 40, right: 170, bottom: 25, left: 40 };
const width = 1000;
const height = 500;
const innerWidth = width - margin.left - margin.right;
const innerHeight = height - margin.top - margin.bottom;
```

The SVG container is appended to a `<div>` and given a `viewBox` so it scales responsively. A `<g>` element is then translated by `margin.left, margin.top` to become the inner chart origin. All chart elements are appended to this group so coordinates are relative to the inner chart area, not the SVG origin.
```js
const svg = d3.select("#line-chart")
  .append("svg")
  .attr("viewBox", `0, 0, ${width}, ${height}`);

const innerChart = svg
  .append("g")
  .attr("transform", `translate(${margin.left}, ${margin.top})`);
```

### Scales and Axes
**Scales** map data values (the domain) to pixel positions (the range). For a line chart showing temperature over time, two scales are needed.

The x-axis uses `d3.scaleTime()` because the domain is a range of dates:
```js
const firstDate = new Date(2024, 0, 1);
const lastDate = d3.max(data, d => d.date);
const xScale = d3.scaleTime()
  .domain([firstDate, lastDate])
  .range([0, innerWidth]);
```

The y-axis uses `d3.scaleLinear()`:
```js
const maxTemp = d3.max(data, d => d.ave_temp);
const yScale = d3.scaleLinear()
  .domain([0, maxTemp])
  .range([innerHeight, 0]);
```
Note the y range is inverted: `[innerHeight, 0]`. This is because SVG y coordinates increase downward, so a value of 0 in data (bottom of chart) must map to `innerHeight` in pixels.

**Axes** are generated by wrapping a scale in an axis generator, then calling it on a `<g>` element appended to the inner chart. The bottom axis must be translated to the bottom of the inner chart:
![[Pasted image 20260401020343.png]]
![[Pasted image 20260401020512.png|518]]
```js
const bottomAxis = d3.axisBottom(xScale)
  .tickFormat(d3.timeFormat("%b"));

innerChart
  .append("g")
  .attr("class", "axis-x")
  .attr("transform", `translate(0, ${innerHeight})`)
  .call(bottomAxis);

const leftAxis = d3.axisLeft(yScale);

innerChart
  .append("g")
  .attr("class", "axis-y")
  .call(leftAxis);
```
Without `.tickFormat(d3.timeFormat("%b"))`, the first tick defaults to showing the year number rather than "Jan". GitHub Copilot can suggest this fix when asked.

### Drawing the Line
![[Pasted image 20260401020608.png]]
The **line generator** (`d3.line()`) takes x and y accessor functions and produces a path string from an array of data objects:
![[Pasted image 20260401020623.png]]
```js
const lineGenerator = d3.line()
  .x(d => xScale(d.date))
  .y(d => yScale(d.ave_temp));

innerChart
  .append("path")
  .attr("d", lineGenerator(data))
  .attr("fill", "none")
  .attr("stroke", "#75485E");
```
`fill` must be set to `"none"` or the path will fill as a closed shape.

### Curve Interpolation
The default curve is `d3.curveLinear`, which draws straight segments between points. Chaining `.curve()` on the line generator changes this behaviour. The available types and their characteristics are:
![[Pasted image 20260401023059.png]]

|Curve|Behaviour|Best for|
|---|---|---|
|`d3.curveLinear`|Straight segments|Exact data representation|
|`d3.curveBasis`|Cubic basis spline, may miss endpoints|Smoothing noisy data|
|`d3.curveBundle`|Straightened spline|Flow diagrams|
|`d3.curveCatmullRom`|Passes through each point, smooth|Temperature, natural curves|
|`d3.curveMonotoneX`|Monotone in x direction|Monotone data|
|`d3.curveStep`|Staircase between points|Step functions, discrete changes|

Always draw scatter points first, then choose a curve type that accurately represents those points rather than distorting them.
```js
const lineGenerator = d3.line()
  .x(d => xScale(d.date))
  .y(d => yScale(d.ave_temp))
  .curve(d3.curveCatmullRom);
```

### Drawing an Area
The **area generator** (`d3.area()`) extends the line concept with a lower boundary (`y0`) and an upper boundary (`y1`), useful for showing temperature ranges or confidence intervals:
![[Pasted image 20260401024248.png]]
```js
const areaGenerator = d3.area()
  .x(d => xScale(d.date))
  .y0(d => yScale(d.min_temp))
  .y1(d => yScale(d.max_temp))
  .curve(d3.curveCatmullRom);

innerChart
  .append("path")
  .attr("d", areaGenerator(data))
  .attr("fill", "#75485E")
  .attr("fill-opacity", 0.2);
```
The area path is appended before the line path so the line renders on top. Setting `fill-opacity` to a low value keeps the shaded band from obscuring other elements.

### Adding Labels
Text labels for line ends are positioned by looking up scale values. Labels are appended directly to `innerChart`:
![[Pasted image 20260401024834.png]]
```js
// Average temperature label at end of line
innerChart
  .append("text")
  .text("Average temperature")
  .attr("x", xScale(lastDate) + 10)
  .attr("y", yScale(data[data.length - 1].ave_temp))
  .attr("dominant-baseline", "middle")
  .attr("fill", "#75485E");
```

![[Pasted image 20260401024745.png]]
`dominant-baseline: "middle"` centres the text vertically on the y coordinate. For labels near the area boundaries, offset from 3 data points before the end (`data.length - 3`) and add pixel nudges:
```js
// Maximum line: nudge label up with + -30 on y
.attr("y", yScale(data[data.length - 3].max_temp) + -30)

// Minimum line: push label down with + 40
.attr("y", yScale(data[data.length - 3].min_temp) + 40)
```

![[Pasted image 20260401025006.png]]
Short SVG `<line>` elements connecting the label to the data point can be appended using `x1/y1` (data point) and `x2/y2` (near label) coordinates.

### Drawing Arcs (Polar Coordinate Charts)
When drawing a simple arc or donut to show a proportion, the chart does not use the Cartesian inner chart group. Instead, arcs use a **polar coordinate system** where position is defined by a radius and an angle rather than x/y.
![[Pasted image 20260401025207.png]]
The SVG group is translated to the centre of the container:
```js
const innerChart = svg
  .append("g")
  .attr("transform", `translate(${pieChartWidth / 2}, ${pieChartHeight / 2})`);
```
All arc elements are then appended relative to this centre origin.

The **arc generator** (`d3.arc()`) is configured with `innerRadius`, `outerRadius`, `padAngle`, and `cornerRadius`. Setting `innerRadius` to 0 produces a pie slice; a positive value produces a donut arc.
![[Pasted image 20260401025310.png]]
```js
const arcGenerator = d3.arc()
  .innerRadius(80)
  .outerRadius(120)
  .padAngle(0.02)
  .cornerRadius(6);
```

Angles are in radians. To draw an arc for a percentage, calculate the angle as follows:
```js
const percent = numberOfDaysWithPrecipitation / numberOfDays;
const angleRad = percent * 360 * Math.PI / 180;
```
Equivalent of $rad = (\frac{percent}{100}) \times 2\pi$

**Each** arc is appended as a `<path>`, passing `startAngle` and `endAngle` to the generator:
```js
innerChart
  .append("path")
  .attr("d", () => arcGenerator({ startAngle: 0, endAngle: angleRad }))
  .attr("fill", "#6EB7C2");

innerChart
  .append("path")
  .attr("d", () => arcGenerator({ startAngle: angleRad, endAngle: 2 * Math.PI }))
  .attr("fill", "#DCE2E2");
```

The centroid of an arc (its visual centre of mass) can be calculated by calling `.centroid()` on the arc generator, which returns `[x, y]`. This is used to position labels:
![[Pasted image 20260401030555.png]]
```js
const centroid = arcGenerator
  .startAngle(0)
  .endAngle(angleRad)
  .centroid();

innerChart
  .append("text")
  .text(d3.format(".0%")(percent))
  .attr("x", centroid[0])
  .attr("y", centroid[1])
  .attr("text-anchor", "middle")
  .attr("dominant-baseline", "middle")
  .attr("fill", "white")
  .style("font-weight", 500);
```
For more than two arc categories, use the pie layout generator (`d3.pie()`) covered in Part 2.

![[Pasted image 20260401030649.png]]
### Key Takeaways
1. The margin convention translates the inner chart group so all coordinates are relative to the drawable area, not the SVG origin.
2. The y scale range is always inverted `[innerHeight, 0]` because SVG y increases downward.
3. Always append the area before the line so the line sits visually on top.
4. Curve type choice is a data fidelity decision: prefer `curveCatmullRom` or `curveMonotoneX` for data that must pass through each point.
5. Arcs use polar coordinates; translate the group to centre before appending any arc paths.

**Remember this:** A line chart in D3 is just a configured generator function applied to your dataset, producing an SVG path string — everything else is axis and label decoration around that single path.

**One analogy to remember it by:** The line generator is like a skilled cartographer handed a list of cities and told to trace a road between them. The scales are the projection system converting real-world coordinates into the map's pixel grid.

## Part 2: Pie, Donut, and Stacked Charts

### What is a Layout Function?
![[Pasted image 20260401135241.png]]
A **layout function** in D3 is a data preprocessing step that takes your original dataset and returns an annotated version of it containing the geometric information needed to draw a specific chart type.

In simple terms: instead of calculating arc angles or stack boundaries yourself, you hand your data to a layout function and it hands back everything D3 needs to draw.

Think of it like: giving a recipe to a sous chef who returns it with every ingredient pre-measured and ready to go. You still do the cooking (drawing), but the prep is done.

Why it matters: complex chart types like pie/donut and stacked bars require non-trivial geometric calculations. Layout functions encapsulate this so you can focus on visual encoding.

The pattern is always: original dataset → layout function → annotated dataset → draw chart.

### Donut and Pie Charts
![[Pasted image 20260401135228.png]]
A **pie chart** or **donut chart** is built in three stages.

**Stage 1: format the data.** The pie generator requires an array of objects each containing a numeric value. For multi-category data read from a CSV, the year column should be filtered out from the column list:
```js
const years = [1991, 2010, 2020, 2024];
const formats = data.columns.filter(format => format !== "year");
// formats = ["beef", "pig", "poultry", "sheep"]
```
![[Pasted image 20260401140220.png]]

Inside a `years.forEach()` loop, the data for a specific year is extracted and reshaped:
```js
const yearData = data.find(d => d.year === year);
const formattedData = [];

formats.forEach(format => {
  formattedData.push({
    format: format,
    production: yearData[format]
  });
});
```

**Stage 2: run the pie generator.** `d3.pie()` calculates `startAngle` and `endAngle` in radians for each category, producing an annotated dataset:
```js
const pieGenerator = d3.pie()
  .value(d => d.production)
  .sort(null); // preserve category order for cross-chart comparison

const annotatedData = pieGenerator(formattedData);
```
![[Pasted image 20260401140457.png]]
`.sort(null)` disables automatic sorting, keeping each meat type in the same angular position across all years for easy visual comparison.

**Stage 3: draw arcs and labels.** The arc generator is configured once and reused. The data-binding pattern uses `.join("g")` so each arc group can hold both a `<path>` and a `<text>`:
```js
const arcGenerator = d3.arc()
  .startAngle(d => d.startAngle)
  .endAngle(d => d.endAngle)
  .innerRadius(60)
  .outerRadius(100)
  .padAngle(0.02)
  .cornerRadius(3);

const arcs = donutContainer
  .selectAll(`.arc-${year}`)
  .data(annotatedData)
  .join("g")
  .attr("class", `arc-${year}`);

arcs.append("path")
  .attr("d", arcGenerator)
  .attr("fill", d => colorScale(d.data.format));
```

Percentage labels hide automatically for slices smaller than 5% using a conditional opacity:
```js
arcs.append("text")
  .text(d => {
    d["percentage"] = (d.endAngle - d.startAngle) / (2 * Math.PI);
    return d3.format(".0%")(d.percentage);
  })
  .attr("x", d => {
    d["centroid"] = arcGenerator
      .startAngle(d.startAngle)
      .endAngle(d.endAngle)
      .centroid();
    return d.centroid[0];
  })
  .attr("y", d => d.centroid[1])
  .attr("text-anchor", "middle")
  .attr("dominant-baseline", "middle")
  .attr("fill", "#f6fafc")
  .attr("fill-opacity", d => d.percentage < 0.05 ? 0 : 1)
  .style("font-size", "16px")
  .style("font-weight", 500);
```
The year label in the donut centre is appended directly to `donutContainer` at position (0, 0), which is the translated group centre.

### Positioning Multiple Donuts
![[Pasted image 20260401145013.png]]
Each donut chart occupies its own `<g>` container translated horizontally. Two positioning strategies exist:

- **Using xScale (band scale):** The x position of each container is mapped from a year value using `d3.scaleBand()`. The scale is declared globally and configured inside a `defineScales(data)` function that runs after data loads. This ensures the band scale domain is populated before any drawing function calls it:
```js
const xScale = d3.scaleBand();
const defineScales = (data) => {
  xScale
    .domain(data.map(d => d.year))
    .range([0, innerWidth]);
};

// In the forEach loop:
const donutContainer = donutContainers
  .append("g")
  .attr("transform", `translate(${xScale(year)}, ${innerHeight / 2})`);
```

- **Using even spacing (GitHub Copilot approach):** Divide the available width by the number of charts:
```js
const chartSpacing = (width - margin.left - margin.right) / years.length;
const chartOffset = chartSpacing / 2;

const donutContainer = donutContainers
  .append("g")
  .attr("transform", `translate(${chartOffset + index * chartSpacing}, ${innerHeight / 2})`);
```

### Colour Scales
![[Pasted image 20260401160102.png]]
An [ordinal colour scale](https://d3js.org/d3-scale/ordinal) maps each category ID to a hex colour. Both the domain and range are derived from the shared `formatsInfo` constant defined in `shared-constants.js`:
```js
const colorScale = d3.scaleOrdinal();

colorScale
  .domain(formatsInfo.map(f => f.id))
  .range(formatsInfo.map(f => f.color));
```
This ensures the colour mapping is consistent across all charts (donut, stacked bar, legend) because they all read from the same `formatsInfo` array.

### Stacked Bar Charts
![[Pasted image 20260401160136.png]]
A **stacked bar chart** follows the same three-stage layout function pattern as the donut chart.
![[Pasted image 20260401160155.png]]
**Stage 1: stack generator.** `d3.stack()` requires a `.keys()` accessor that names the categories to stack. It returns an annotated dataset where each series contains lower (`d[0]`) and upper (`d[1]`) boundary values for every data point:![[Pasted image 20260401160719.png]]
![[Pasted image 20260401160302.png]]
```js
const stackGenerator = d3.stack()
  .keys(formatsInfo.map(f => f.id))
  .order(d3.stackOrderDescending); // largest values at bottom for readability

const annotatedData = stackGenerator(data);
```
![[Pasted image 20260401160805.png]]
`d3.stackOrderDescending` places the largest series at the base of each bar, so the dominant category (pork) anchors the bottom edge and comparisons across years are easier to read.

**Stage 2: y scale.** The y scale's maximum is the highest upper boundary in the last series of the annotated data (the top of the tallest bar):
```js
const maxUpperBoundary = d3.max(
  annotatedData[annotatedData.length - 1],
  d => d[1]
);

const yScale = d3.scaleLinear()
  .domain([0, maxUpperBoundary])
  .range([innerHeight, 0])
  .nice();
```

**Stage 3: draw rectangles.** Each series in `annotatedData` is iterated with `forEach`. Inside each series, the data-binding pattern creates one `<rect>` per year. The rectangle's y position is the scaled upper boundary, and its height is the difference between the scaled lower and upper boundaries:
```js
annotatedData.forEach(series => {
  innerChart
    .selectAll(`.bar-${series.key}`)
    .data(series)
    .join("rect")
    .attr("class", d => `bar-${series.key}`)
    .attr("x", d => xScale(d.data.year))
    .attr("y", d => yScale(d[1]))
    .attr("width", xScale.bandwidth())
    .attr("height", d => yScale(d[0]) - yScale(d[1]))
    .attr("fill", colorScale(series.key));
});
```

The class name `bar-${series.key}` is unique per series so elements from different series are not accidentally merged during data joins.

**Axes** are added using `d3.axisBottom` and `d3.axisLeft`. For the x-axis, `tickValues(d3.range(1991, 2024, 5))` shows only every fifth year to avoid overcrowding:
```js
const bottomAxis = d3.axisBottom(xScale)
  .tickValues(d3.range(1991, 2024, 5))
  .tickSizeOuter(0);
```

### Proportional Stacked Bar Chart
![[Pasted image 20260401161016.png]]
To convert an absolute stacked bar chart into a proportional (100%) version, two changes are needed: add `.offset(d3.stackOffsetExpand)` to the stack generator, and change the y scale domain to `[0, 1]` with percentage tick formatting:
```js
const stackGenerator = d3.stack()
  .keys(formatsInfo.map(f => f.id))
  .order(d3.stackOrderDescending)
  .offset(d3.stackOffsetExpand); // normalises each bar to 100%

const yScale = d3.scaleLinear()
  .domain([0, 1])
  .range([innerHeight, 0])
  .nice();

const leftAxis = d3.axisLeft(yScale)
  .tickFormat(d3.format(".0%"))
  .ticks(5)
  .tickSizeOuter(0);
```
`d3.stackOffsetExpand` rescales all values so each bar sums to 1, making proportional change visible across time even when absolute totals are growing.

### File Structure and Script Loading Order
Because `defineScales` must run before any drawing function, and because the drawing functions depend on shared constants, the `index.html` script loading order matters:
```html
<script src="https://d3js.org/d3.v7.min.js"></script>
<script src="js/shared-constants.js"></script>
<script src="js/scales.js"></script>
<script src="js/donut-charts.js"></script>
<script src="js/stacked-bars.js"></script>
<script src="js/legend.js"></script>
<script src="js/load-data.js"></script>
```
`load-data.js` is last because it triggers the data load and then calls `defineScales`, `drawDonutCharts`, `drawStackedBars`, and `addLegend` in sequence inside the `.then()` callback.

### Donut vs Stacked: When to Use Each

|Chart type|Best for|Limitation|
|---|---|---|
|Donut/Pie|Proportions at a single point in time, 4-5 categories max|Poor for tracking change; too many slices become illegible|
|Stacked bar (absolute)|Total volume and composition over time|Hard to compare non-bottom segments|
|Stacked bar (proportional)|Relative share change over time|Hides absolute volume changes|

### Key Takeaways
1. Layout functions (`d3.pie()`, `d3.stack()`) are data preprocessing steps that return annotated datasets with the geometric values needed to draw the chart.
2. `.sort(null)` on the pie generator keeps category order consistent across multiple donut charts for fair visual comparison.
3. For stacked bars, rectangle height is `yScale(d[0]) - yScale(d[1])` because the y scale is inverted.
4. `d3.stackOffsetExpand` converts a stacked bar to a 100% proportional chart with no other changes to the drawing code.
5. Script loading order in `index.html` is critical: shared constants and scales must load before drawing functions, and the data loader must load last.

**Remember this:** Pie and stack charts always follow the same three-step pattern: format data, run the layout generator to annotate it, then bind the annotated data to SVG elements using the drawing generator.

**One analogy to remember it by:** The pie and stack generators are like surveyors who walk your raw land data and return a blueprint with every boundary staked and measured. You are the builder who follows the blueprint to put up walls (paths and rectangles), but you never had to do the surveying yourself.
